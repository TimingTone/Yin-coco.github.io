Data Structure Tree

Binary Tree

Posted on 2019-10-04 • In Data Structure , Tree

Although the title here is BT, we hava to know the definition of Tree first.

Vector and List belong to linear data structures, but Tree isn’t one kind of linear structure in the strict sense.

Tree

Rooted Tree

The Tree structure is nothing but a kind of special graph. That is to say, it can be regarded as a set of objects where some pairs of objects are connected by lines. If some of this objects have this relation, then we introduce an edge. Then all the objects that can be defined relation between them will be called vertex.

Vertex is different from the Node we use in computer programming.

The Tree we defined here has following features:

  • We will assign a special Node as root. This Tree will be called Rooted Tree.

  • By connecting each other, a small rooted tree that can become a larger rooted tree step by step.

  • Compared to this larger tree, every small tree of its parts is called subtree. These subtrees will have a new root r.

Ordered Tree

Under the foregoing conditions, the root of a subtree is called child. On the other hand, the new root r is the father of these nodes. For any node $r_i$, the number of its children is called its degree.

In every single tree, the number of all edges equals to the whole degrees of all the nodes. At the same time, it equals the number of nodes minus 1.

The $r_i$ is called sibling by each other. We will give all the children of one node a serial number to make them hace a order in this sense. All the trees that have been defined specific order in the siblings also are called ordered tree.

Representation of the Tree

Let’s discuss how to express a tree logically in a computer

Interface

Now it is too early to talk about the implementation of these interfaces. We should learn to describe a tree logically first.

Left child right brother representation

The most efficient logical method to represent a tree is firstChild and brother representation. We can find that two pieces of information are enough for one node to express downward reference.

  • Portrait: firstChild()

  • Landscape: nextSibling()

That means we just need to care about its first child and next sibling for any node. In the following, we will introduce binary tree, and when we represent all the tree with a binary tree, we will use this representation again.

It’s funny that although the binary tree is a special subset for the universal set tree, it is enough to represent all the trees under some constraints. The principle behind such a method is mostly based on the first child and brothers methods.

Binary Tree

Overview

Binary Tree: A tree in which the degree of each node does not exceed 2, up to two branchs. That’s why it’s called a binary tree.

Therefore, there is a maximum of two children per node in any one part, so you may wish to disguish them by left and right. Similarly, the corresponding two subtrees can also be called as left and right. We call the left child, the left subtree relative some node r, or the right child, the right sub-tree relative to r.

Please note that here we have actually implied the orderliness of the tree. In general, our habit is to think that the left is first and the right is behind.

What’s interesting, however, is that we’ll soon see that the binary tree is sufficient to decribe all the trees under the premise that the roots and orderliness can be guaranteed.

The k-th equivalence class, that is, all nodes with depth K, will not exceed 2 to the power of k.

The overall size of the entire tree meets such as an upper bound, that is, twice as much as the last level of the last equivalence class. Because for a tree with the height of h, there is at least one node per layer. If this is the case, the tree will be retrograded into a single chain consisting of one node at each level.

Conversely, except for leaf nodes, if all nodes have a degree of 2, then equivalence class will reach its maximum saturation, thus constituting a special tree, known as full tree.

Description of Multi-fork Tree

How to describe a multi-branch tree, that is, a tree in general, with a binary tree?

You may say, “Apparently, this is not possible because you seem to be using a special case to describe everything.” However, it’s very interesting that this turned out to be viable.

This is not without conditions. Only two: Rooted and Orderly. In other words, all rooted and ordered trees can be represented by a binary tree description and then implemented.

The method of implementation is eldest son and brother representation.

Traversal

Conversion Strategy

Now we have learned how to implement a binary tree. So starting from this section, we will focus on the algorithms of the binary tree. As with many previous issues, binary tree related algorithms often need to access elements one by one in a certain order. However, binary tree isn’t one kind of linear structure in a strict sense. At least it’s a semi-linear structure.

From now on, a strategy for solving problems I will introduce to you, that is, instead of building from the wheels, you have to be good at using the results of previous works. As far as the binary tree we talk is concerned, that is how we can transform this semi-linear structure into the linear structure.

Specifically, it is to define some kind of explicit linear order among all the nodes of the binary tree according to some pre-agreed principle.

The specific conversion method is the traversal. In fact, this strategy will be adopted through fundamental data structure. For example, the graph structure is a typical nonlinear structure.There, we’ll also deal with the problem of how to transform a nonlinear structure into a relatively well-researched semi-linear structure. Our approach is still traversal.

Traversal Rules

In order to be able to access each node in the tree once in an order, any local sub-tree can be divided into three parts:

  • The root of the tree V

  • The left sub-tree L

  • The right sub-tree R

As long as the access to the root node and the order of traversal of the left and right sub-trees can be clearly determined, it will inevitably lead to a clear linear order as a whole.

Indeed, as long as we follow a principle everywhere, we can export several typical traversal algorithms:

  • preorder traversal

  • In-order traversal

  • Post-order traversal

The difference between them lies in the order in which the local root nodes are accessed.

Preorder In-order Postorder
V | L | R L | V | R L | R | V

Preorder Traversal

Here we provide a eventual implementation strategy, but I highly recommend some other realizations from here to help you understand the eventual strategy.

Our strategy is mainly based on following diagram:

Let’s assume that the length is d and ends at $L_d$. Then, each node along the way should also have its own right child $R_i$ and right sub-tree $T_i$. Although it’s possible that the right child or the right sub-tree does not exist at all, we might as well fold it up and draw it in a uniform manner. You can easily convince youself that any binary tree can be abstractly represented as such a chain descending along the left chain, while the other parts are respectively classified into the right sub-tree of the nodes along the left chain.

After such an abstract thinking, we can clearly see the order and context of the entire sequence of preorder traversal. It can indeed be seen that, in any such sub-tree part, our access order is to first access each node in turn along such a left branch. Such a process will inevitably end at the $L_d$. Then, from a macroscopic point of view, we should turn turn the direction and go from bottom to top in order to traverse the series of the right sub-trees previously defined.

OK, Let’s summarize it again. The whole process is divided into two phases.

  • The first step is to access the nodes along the left side chain from top to bottom.

  • Then turn it upside down and traverse each right sub-tree on each level in turn.

This is the macroscopic process through which we traverse the entire sequence. And our algorithm is devised along this line of thought.

In-order Traversal

This is also an eventual conception:

Similar to the preorder traversal, any binary tree can be abstractly defined as shown in the diagram. We can find that in any one part, when the control is transfered to the current root node, this node does not immediately accept the access, but, figuratively, it will yield this control to its left child and it won’t go deep into its right sub-tree until the left child is accessed. And then, only after the right sub-tree is traversed, the control will return to the previous root node. And only at this time, this root node will readily accept the access.

Such a patial order pattern, if translated into such a whole tree, will inevitably lead to such an overall order. The raversal process of the whole tree can be devided into several stages.

So how can such a concept be specifically implemented as code. We have noticed that there is some kind of reverse order in such a process. Our initial starting point is at the root node. However, it is the end node of the left-side chain of it that is first accessed. If we say that the entire yielding process is top-down, then it’s not difficult for us to find the order in which nodes are actually accessed is generally a bottom-up process.

Without doubt, we should be able to adopt a structure that supports last-in, first-out. That’s right, it’s stack.

Hierarchical Traversal

Order

In this section, we will talk about the last typical binary tree traversal algorothm, namely hierarchical traversal.

Recall that, at beginning of this blog, we had agreed that we always adopt the rooted ordered trees. In other words, any binary tree discuss here is assigned a special node, which we call the root node. With that, all nodes can be divided into several equivalent classes per the depth in the vertical direction. Therefore, we can think that the so-called rootedness actually corresponds to order of such a vertical direction.

So, further on, how do we define the order of all the nodes that are also within the same equivalence class at the same depth. As what we discussed, all peer nodes can indeed be separated. For example, for a binary tree, the relative order between sibling nodes can be given through the explicit definition of left and right. So we can think that the so-called orderliness also just provides an order along the horizonal direction. According to the order of the vertical directon and the horizonal direction, we totally can define the overall order among all the nodes and then traverse it.

Specifically. we will access each node in the tree top-down from left to right in each layer. The traversal strategy and process defined as such is hierarchical traversal.

Implementation of HT

How exactly should such a traversal strategy be implemented as code specifically?

What kind of data structure do we need to use for this?

Now let’s recall the three typical traveral strategies decribed earlier: whether a preorder, a inorder or a postorder, none can guarantee that all nodes will be accessed strictly in order of depth. In these three traversal strategies, there are phenomena in which descendants are accessed before the ancestors, which is the so-called reverse order. For this reason, as we have seen, when imlementing these traversal strategies, we need to rely on the stack structure whether implicit or explicit.

Conversely, in the hierarchical traversal, all the nodes that we have just defined will be in accessed in strict descending order, from high to low. In other words, the order is strictly met here.

Therefore, it is easy to understand that the kind of data structure that is completely symmetrical to the stack in such a situation, that is, the queue, will show its full potential.

call-by-key

Concretely, each of BST data items has its own key code as a feature to distinguish from each other.

So the first important point is call-by-key, which meets following conditions.

  • Be able to compare in size with each other

  • Support equal comparison

We might as well assume that data items in the entire data set have been uniformly represented and implemented as a form of Entry.

Order

BST is Binary Tree first.Its fundamental element is node.And each of which contains an entry. Besides,each entry refers to only one key. For the convenience of statement,as long as it won’t be confusing,the three words:Node, Entry, and Key, refer to the same thing in the following introductions.

Let’s summarize the character of BST:

A BST is in order everywhere.

  • The specific order is that as for a node V, if it has left descendants, the values of them won’t be bigger than the value of node V.

  • Symmetrically, if it has right descendants, the values of them won’t be smaller than value of node V.

Monotonicity

Although the rule of order defined before is just a local feature, it is exactly this feature can elicit a global feature.

Specifically, it’s so-called monotonicity.

Thus, in summary, the feature of a BST is that it follows the rules of order in microstructure, and is monotonic in the whole.
